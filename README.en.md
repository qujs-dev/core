# Qu.js

- Project website: https://qujs.ru/
- Documentation: https://qujs.ru/api/
- License: MIT (see `LICENSE`)

Qu.js is a utility‑oriented JavaScript library that provides independent tools for solving common web‑development tasks: asynchronous loading of scripts and styles, event management, waiting for DOM readiness, loading external libraries, and wiring modules together.

It does not impose any architecture and simply offers convenient tools that can be used separately or together. Thanks to its Promise‑based API and the queue system (`Que`), you can build dependency chains and ensure that code runs at the right moment — whether it is page load, an element appearing in the viewport, or a certain variable becoming available.


# Key features

- **Task queue** (`Que`) — defer execution of functions until one or more events have occurred (DOMReady, library loaded, custom signal). Supports `multi` (all events), `series` (sequential events), and `ordered` (strict order) modes.
- **Asset loader** (`loadAssets`) — load scripts, styles, and images with order control and caching. You can set a timeout and force reload.
- **Event management** (`on`, `off`, `trigger`) — subscribe to events with delegation, element collections support, and the ability to aggregate handler responses.
- **Variable watcher** (`def`) — react to the moment when a global variable or its nested property becomes defined.
- **Modular architecture** — the core library acts as a kernel, and additional modules (for example, Lazy for lazy loading) are plugged in as plugins via `Qu.lib()`.
- Plus a few utilities for working with the DOM and scrolling.


# Philosophy


This is just a collection of tools originally designed to be used with `async`/`defer` so that scripts do not block page loading — this approach turned out to be convenient in my projects. Part of the code is generated by neural networks and then refined by hand, and the utilities target modern browsers.

If this approach resonates with you, feel free to use it.


# Quick start


## Loader (required, very first script) loader.js

```javascript
if (!window.Que) {
    (function () {
      var originalDispatch = window.dispatchEvent;
      var fired = window._QueFired || (window._QueFired = {});
      var queue = window._QueQ || (window._QueQ = []);
    
      window.dispatchEvent = function (event) {
        if (event && event.type) {
          fired[event.type] = (fired[event.type] || 0) + 1;
        }
        return originalDispatch.call(window, event);
      };
    
      window.Que = function (callback, ev, onceOrOptions, mode, useCache) {
          if (window.Qu && typeof window.Qu.when === 'function') {
              return window.Qu.when(callback, ev, onceOrOptions, mode, useCache);
          }
          queue.push([callback, ev, onceOrOptions, mode, useCache]);
      };
    })();
};
```

This code must run before any `Que(...)` calls. That’s why it should be included first, synchronously, in `<head>` before any other scripts if you want to track variable and event readiness. P.S. you can skip it — Qu.js has a built‑in fallback — but if you don’t load it first, you won’t be able to correctly track variables/events.


## Libraries / application modules

```html
<link rel="stylesheet" href="/qu/core/css/qu.min.css"
    media="print" onload="this.media='all'">

<script src="/qu/lazy/lazy.min.js" async></script>
<script src="/qu/core/qu.min.js" defer></script>
```

or via CDN, cached for one year:

```html
<link rel="stylesheet" href="https://cdn.jsdelivr.net/gh/qujs-dev/core@1.0.0/css/qu.min.css"
    media="print" onload="this.media='all'">

<script src="https://cdn.jsdelivr.net/gh/qujs-dev/lazy@1.0.0/lazy.js" async></script>
<script src="https://cdn.jsdelivr.net/gh/qujs-dev/core@1.0.0/qu.min.js" defer></script>
```

or latest versions via `@main` (7‑day cache):

- https://cdn.jsdelivr.net/gh/qujs-dev/lazy@main/lazy.min.js  
- https://cdn.jsdelivr.net/gh/qujs-dev/core@main/qu.min.js  

Additional libraries are recommended to be loaded with `async` in any order. The Qu.js core itself should use `defer` so it does not block rendering.



# Recommendations

Using a large number of async/defer scripts requires HTTP/2 (or HTTP/3).

CDN servers have HTTP/2–3 enabled by default, but free CDNs are not always stable — that’s why a fallback to a local file is a sensible safeguard.

```html
<link
  rel="preload"
  as="style"
  href="https://cdn.jsdelivr.net/gh/qujs-dev/core@1.0.0/css/qu.min.css"
  onload="this.rel='stylesheet'"
  onerror="this.onerror=null;this.href='/qu/core/css/qu.min.css';this.rel='stylesheet'"
>

<!-- preferable, but without the fallback -->
<link
    href="https://cdn.jsdelivr.net/gh/qujs-dev/core@1.0.0/css/qu.min.css"
    rel="stylesheet"
    media="print"
    onload="this.media='all'; this.onload=null;"
>

<script
  src="https://cdn.jsdelivr.net/gh/qujs-dev/core@1.0.0/qu.min.js"
  onerror="this.onerror=null;this.src='/qu/core/qu.min.js'"
  async>
</script>
```

On HTTP/1.1, browsers limit the number of simultaneous connections per domain, and async/defer scripts will end up queued. On HTTP/2–3, multiplexing allows all resources to be downloaded in parallel without such delays.

If the server runs on HTTP/1.1 and the Qu.js library files are physically hosted on it, using async/defer will not reduce the number of parallel downloads; in that case, it is often better to bundle scripts into a single file (an old‑school but reliable approach).


# Code examples

`Que` — by default reacts to DOM readiness.

```javascript
Que(function() {
    console.log('DOM ready');

    Qu.def(['fancybox', 'jQuery'], function(vars) {
        console.log('All libraries loaded!');
        Qu.trigger(window, 'app:ready', { detail: { fancybox, jQuery } });
    });
});
```

Specify your own event — `Que` will fire when it is dispatched.

Provided that the mini‑loader is already connected, you can place `Que` calls anywhere on the page or in any file — they will still work, because the loader intercepts calls and buffers them until the main Qu.js core is loaded.

```javascript
Que(function() {
    console.log('✅ App ready!');
    // todo
}, 'app:ready');
```

You can wait for several events at once, in sequence, or in strict order (see API):

```javascript
Que(function() {
    console.log('✅ All steps completed from the start');
    // todo
}, ['step1:complete', 'step2:complete', 'step3:complete'], false, 'ordered');
```

In this example, after each successful run the order is reset, and a new series of events in the same order is awaited. The `false` parameter means the callback will run every time the sequence repeats, not just once.


### Initializing a library written in Qu style

```javascript
Que(function() {
    Qu.libs(['Lazy'], { autoInit: true, initParams: {} }).then(function(loadedLibs) {
        console.log('Lazy ready');
    });
});
```

or:

```javascript
Que(function() {
    // If Lazy is already loaded — the callback will fire immediately
    console.log('Lazy ready!');
    Qu.Lazy.init();
}, 'qu:lazy:loaded');
```


## Example of extending an object at each nesting level

```javascript
Que(function() {
    console.log('1️⃣ DOM ready');

    const user = { name: 'Player' };

    setTimeout(() => {
        Qu.trigger(window, 'step1:done', { detail: user });
    }, 300);

    Que(function(Qu, data) {
        console.log('2️⃣ Received:', data);
        data.age = 13;

        setTimeout(() => {
            Qu.trigger(window, 'step2:done', { detail: data });
        }, 300);

        Que(function(Qu, data) {
            console.log('3️⃣ Received:', data);
            data.city = 'Suncity';

            setTimeout(() => {
                Qu.trigger(window, 'step3:done', { detail: data });
            }, 300);

            Que(function(Qu, data) { 
                console.log('4️⃣ Final object:', data);
                console.log('Name:', data.name);
                console.log('Age:', data.age);
                console.log('City:', data.city);
            }, 'step3:done');

        }, 'step2:done');

    }, 'step1:done');

}, 'qu:dom'); // you can omit 'qu:dom' (it’s the default) or use another event
```

> 1️⃣ DOM ready  
> 2️⃣ Received: { name: 'Player' }  
> 3️⃣ Received: { name: 'Player', age: 13 }  
> 4️⃣ Final object: { name: 'Player', age: 13, city: 'Suncity' }  
> Name: Player  
> Age: 13  
> City: Suncity
